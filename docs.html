Design Documentation

How This Was Built

A record of the design decisions, technical architecture, and user experience principles that shaped this cinematic tourism exhibition — from concept to code.

01

Concept & Information Architecture

The core challenge was avoiding the twin traps of the genre: the film database (comprehensive but inert) and the tourist guide (useful but shallow). The solution was to impose a narrative layer over the location data — organizing the same factual content into two distinct interpretive journeys.

This dual-narrative structure required an information architecture where the same location object could carry different content, different ordering, and a different atmospheric register depending on the selected narrative. Rather than duplicate location pages, the application uses a single data model with narrative-specific text attached to each location entry.

The site is organized into six pages: Cover (landing/selection), Map (spatial overview), Explore (narrative experience), About, Documentation, and Disclaimer. Navigation between pages is linear and hierarchical, but the Explore page provides a self-contained app-like experience with its own internal navigation.

Data Model

All content is stored in a single data.js file as structured JavaScript objects. This separates content from presentation and allows the UI to be driven entirely by data. Location entries contain per-narrative text arrays, making switching instantaneous.

State Management

Application state (active narrative, active location index) is persisted in localStorage, allowing users to navigate between pages without losing their position. A centralized App module exposes a clean API for state reads and writes.

02

UX Design Principles

The design prioritizes immersion over information density. Each screen presents a single location with generous space for the narrative text — resisting the temptation to show everything at once. The user is guided through the experience sequentially, like chapters of a book or scenes of a film.

Three interaction modes were designed for different user contexts:

  • Planning mode (home): Use the Cover page to select a narrative, the Map to get spatial orientation, then the Explore page to read through locations in sequence. Keyboard arrow keys support quick navigation.
  • On-site mode (field): Use the Map page to identify nearby locations and navigate to the Explore page for contextual narrative. The chapter strip allows quick jumping between thematic sections without sequential browsing.
  • Research mode (desktop): The Documentation and About pages support a scholarly reading of the project's methodology. All location texts include "Narrative Notes" — analytical commentary that goes beyond the cinematic surface.

Navigation Patterns

Previous/Next buttons for linear traversal. Dot indicators for quick overview and direct access. Chapter strip for semantic jumping. Keyboard arrows (← →) supported on the Explore page. Narrative toggle preserves the current location index where possible.

Progressive Disclosure

Information is layered: film tag and location name first, then scene context, then the atmospheric quote, then the full description, then the analytical narrative note. Users can read at whatever depth suits them without being overwhelmed on arrival.

Responsive Layout

The Explore page uses a two-column split layout on desktop (visual left, narrative right) that stacks vertically on mobile. The Map sidebar adapts to a horizontal strip on small screens. Touch targets meet WCAG 2.1 minimum size requirements.

Narrative Transition

Switching between narratives triggers a subtle fade-transition on the content panel (opacity + translateY), providing temporal feedback that the content has changed without being distracting. The transition takes 350ms — fast enough to feel responsive, slow enough to read as intentional.

03

Visual Design System

The visual language draws from cinematic conventions: dark projection-room backgrounds, high-contrast typography, and a color palette inspired by the visual grammar of the spy film genre. The design system is built on CSS custom properties for consistency and easy theming.

Typography

Playfair Display

Headings & Titles

A transitional serif with strong optical contrast. Its literary associations and cinematic weight make it ideal for location names, section headers, and atmospheric quotes. Used at weights 400 (regular italic) and 600–700 (display).

Inter

Body Text & UI

A humanist sans-serif optimized for screen readability. Its clean geometry balances the ornamental quality of Playfair Display. Used at weights 300–400 for body text, 500–600 for UI labels and navigation.

JetBrains Mono

Coordinates & Data

Used for GPS coordinates, film years, and technical data. Its monospaced character creates a deliberate contrast between the narrative text and factual information, reinforcing the dual nature of the experience.

Type Scale

Fluid type sizes using clamp() for responsive scaling without media query breakpoints. Base unit: 16px. Scale: 0.62rem (labels) → 0.9rem (body) → 4.5rem (hero title).

04

Technical Architecture

The project is built as a multi-page static website using vanilla HTML, CSS, and JavaScript — no build tools, no frameworks, no server required. This choice reflects the course context (accessibility, comprehensibility, portability) and the content (a static exhibition does not require dynamic server infrastructure).

File Structure

index.html ← Cover/landing map.html ← Interactive map explore.html ← Narrative explorer about.html ← About page docs.html ← This page disclaimer.html ← Disclaimer css/style.css ← Design system js/data.js ← All content data js/app.js ← Shared state/logic js/map.js ← Leaflet map js/explore.js ← Explore logic

Dependencies

Leaflet.js 1.9.4 — interactive map library (via CDN, no API key required).

CARTO Dark Matter — free dark tile layer compatible with OpenStreetMap data, no registration required.

Google Fonts — Playfair Display + Inter + JetBrains Mono (via CDN).

No other external dependencies. No npm, no bundler, no framework.

Semantic HTML

All pages use semantic HTML5 elements: <nav>, <main>, <section>, <article>, <header>, <footer>, <blockquote>. ARIA attributes provide accessibility context for interactive components.

CSS Architecture

Single stylesheet with clear section comments. Design tokens in :root CSS custom properties. Mobile-first media queries. No preprocessor required. Utility classes for animation and spacing. Component classes follow a BEM-influenced naming convention without strict enforcement.

05

Content Strategy

Each location entry contains four distinct content layers, each serving a different user need and register:

  • Atmospheric Quote — a single evocative sentence in italics. Functions as a threshold: introduces the mood of the location before the reader commits to the full text. Short enough to read in passing.
  • Narrative Description — 100–150 words. Describes the scene from within its narrative context, as if experienced from the film's perspective. Written in present tense for immediacy. References specific film sequences with precise geographic detail.
  • Narrative Note — 60–100 words. An analytical commentary that steps outside the film's perspective to provide historical, geographic, or cinematic context. Suitable for readers with scholarly interests.
  • Film Tag & Coordinates — factual anchors always visible on the visual panel. The GPS coordinates connect the cinematic text to the physical reality of the location, supporting on-site use.

06

Accessibility Considerations

The project targets WCAG 2.1 Level AA compliance. Key measures include:

  • ARIA labels on all interactive elements, landmark regions, and live regions. Navigation buttons have descriptive aria-label attributes.
  • Keyboard navigation — full keyboard operability on the Explore page, including arrow-key location traversal and focus management.
  • Color contrast — primary text on dark backgrounds exceeds 4.5:1 ratio. Muted text is used only for supplementary information, never for primary content.
  • Live regions (aria-live="polite") on the narrative content and navigation badge, ensuring screen reader users are notified of content changes without interruption.
  • Semantic structure with proper heading hierarchy (h1→h2→h3) on all pages.
